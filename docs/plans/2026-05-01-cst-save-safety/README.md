# CST: согласованность дерева и диска при сохранении

## Контекст

- Были случаи порчи файла после CST-команд.
- Цель: не писать на диск при сомнительном состоянии; явная ошибка в ответ и в лог.

## Тезисы (индекс)

- **Источник правды перед записью** — сериализованное CST-дерево (`module.code`), не фрагменты файла.
- **Снимок при загрузке** — хэш (и при необходимости длина/версия) исходного текста файла в момент `cst_load_file` / `reload_tree_from_file`; хранить на объекте дерева в памяти.
- **Перед записью** — прочитать файл с диска; если не совпадает со снимком → **не писать**, код ошибки вроде `FILE_CHANGED_SINCE_LOAD`, лог с путём и `tree_id`.
- **Replay-проверка (самосогласованность сервера)** — на копии модуля/дерева из снимка текста повторно применить те же `TreeOperation`, что в запросе; сериализация должна совпасть с текущим рабочим деревом; иначе **не писать**, код вроде `CST_REPLAY_MISMATCH`, лог (критично: баг в `modify_tree` или в конвертации ops).
- **После атомарной записи** — прочитать файл, сравнить с тем, что собирались записать; при расхождении — критический лог, сценарий отката из бэкапа (уже есть BackupManager) по существующей политике.
- **Общая точка** — вызовы проверок из `save_tree_to_file`, чтобы `cst_save_tree` и save-ветка `cst_modify_tree` не расходились.
- **Превью** — не менять снимок; после успешного `preview` дерево откатывается (уже есть логика отката в modify); снимок остаётся от последней успешной синхронизации с диском.
- **Новые файлы** — отдельное правило: нет «старого» снимка → проверка только replay + пост-запись (или только валидация), без disk-compare.
- **Документация MCP** — описать новые коды ошибок. **`force_overwrite` — вне объёма первой фазы; не реализовывать.**
- **Вне объёма первой фазы** — `query_cst` / `compose_cst_module` (другой путь записи): отдельный подплан. **Сюда же относятся:** `PythonFileHandler` из `code_analysis/core/file_handlers/python_handler.py` и команды `universal_file_save/replace/delete` для `.py` — они используют `run_ops_mode` (`compose_cst_ops_flow.py`), который не проходит через `save_tree_to_file`.

## Глоссарий (коротко)

| Термин | Смысл |
|--------|--------|
| **Снимок диска** | Хэш (и длина) текста файла на момент `load_file_to_tree` / `reload_tree_from_file`. |
| **Replay** | Повторное применение того же списка `TreeOperation` к тексту с диска и сравнение сериализации с рабочим деревом. |
| **`tree_id`** | Идентификатор дерева в памяти сервера; клиент передаёт его в modify/save. |
| **`SuccessResult` + `save_error`** | Текущий паттерн в `cst_modify_tree` при сбое save; новые коды верификации должны быть различимы в `details`. |
