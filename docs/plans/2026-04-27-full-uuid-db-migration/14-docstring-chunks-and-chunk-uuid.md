# Step 14 — Docstring chunks and `chunk_uuid` after UUID migration

## Goal
Update docstring chunk persistence so `code_chunks.id`, `code_chunks.file_id`, `code_chunks.project_id`, and optional entity references use UUID identifiers, while preserving the separate stable `chunk_uuid` business key.

## Architecture constraint
The chunk path must follow the project layering:

```text
command/application code -> universal DB/driver contract -> specific driver -> specific DB
```

`DocstringChunker` and application code may produce domain data and UUID values, but PostgreSQL-specific upsert SQL, casts, placeholder normalization, and adapter expectations belong in the PostgreSQL-specific driver branch.

## Current code checked
`code_analysis/core/docstring_chunker_pkg/docstring_chunker.py` currently builds chunk UUIDs from a string that includes `file_id`:

```python
uuid_name = (
    f"{file_id}:{it.ast_node_type}:{it.line}:{it.source_type}:"
    f"{chunk_index}:{text_sig}"
)
chunk_uuid = str(uuid.uuid5(uuid.NAMESPACE_URL, uuid_name))
```

After full migration, `file_id` will be a UUID string. This changes generated `chunk_uuid` values if the same formula is used with UUID file IDs instead of old integer IDs.

## Required design decision
Choose exactly one `chunk_uuid` migration strategy before implementation.

### Option A — Preserve existing chunk_uuid values during migration
- Keep old `chunk_uuid` values in migrated `code_chunks` rows.
- For future chunks after migration, define whether the formula uses UUID file_id.
- Risk: same file re-chunk after migration may generate different `chunk_uuid` unless compatibility logic preserves old integer mapping.

### Option B — Versioned chunk UUID formula
- Add formula versioning, for example `chunk_uuid_version` or a namespaced UUID input prefix.
- Old rows keep existing values.
- New rows use a new formula based on UUID `file_id`.
- Upsert/conflict logic must handle both old and new rows intentionally.

### Option C — Make `chunk_uuid` independent from DB row id
- Generate chunk_uuid from stable file business identity plus content/position, not from internal row id.
- This is a larger design change and must be tested against file moves/reindexing.

Recommended first migration path:
- Preserve existing `chunk_uuid` values for migrated rows.
- Use versioned formula for new rows if old integer mapping cannot be retained safely.

## Required schema/write changes
1. `code_chunks.id` becomes UUID primary key.
2. `code_chunks.file_id` becomes UUID FK to `files.id`.
3. `code_chunks.project_id` must be validated as UUID-shaped and must remain a denormalized FK/reference to `projects.id`.
4. `code_chunks.class_id`, `function_id`, `method_id` become UUID nullable FKs.
5. `chunk_uuid` remains unique and separate from `code_chunks.id`.
6. `code_chunk_sql.py` must include explicit `id` in the insert/upsert if `id` is not generated by PostgreSQL default.
7. `CODE_CHUNK_UPSERT_PARAM_ORDER` must be updated if `id` is added to bound params.
8. PostgreSQL adapter normalization/translation tests must be updated if bound parameter order changes.
9. `postgres_run` adapter expectations must be updated if SQL placeholders, casts, or returned columns change.
10. PostgreSQL upsert remains in the driver/schema layer, not in `DocstringChunker`.

## Layering requirements
- `DocstringChunker` builds chunk domain data and IDs, but must not contain PostgreSQL `ON CONFLICT` SQL.
- `code_chunk_sql.py` owns portable code chunk upsert shape.
- PostgreSQL driver/adaptation owns PostgreSQL-specific SQL.
- Migration code owns old-int to new-UUID mapping.
- SQLite compatibility logic, if needed, must not change PostgreSQL upsert design.

## Must not do
- Do not drop `chunk_uuid`.
- Do not confuse `chunk_uuid` with `code_chunks.id`.
- Do not use `code_chunks.id` as FAISS vector id without Step 15 decision.
- Do not infer file ownership from path.
- Do not silently change `chunk_uuid` formula without tests and migration notes.
- Do not put PostgreSQL-specific SQL in chunker code.
- Do not add `id` to upsert params without updating `CODE_CHUNK_UPSERT_PARAM_ORDER`, adapter normalization tests, translation tests, and `postgres_run` expectations.

## Tests required
1. Existing migrated chunk keeps its old `chunk_uuid`.
2. New chunk insert creates UUID `code_chunks.id`.
3. `code_chunks.file_id` references UUID `files.id`.
4. `code_chunks.project_id` is UUID-shaped and matches the owning project.
5. `code_chunks.class_id`, `function_id`, and `method_id` are UUID-shaped when present.
6. Upsert by `chunk_uuid` still updates the expected row.
7. Wrong-length chunk upsert params fail clearly.
8. Re-chunking the same docstring after migration follows the chosen versioning strategy.
9. `CODE_CHUNK_UPSERT_PARAM_ORDER` matches actual bound params exactly.
10. PostgreSQL adapter norm/translation tests cover the final chunk upsert SQL.
11. `postgres_run` adapter expectation tests cover final placeholders/casts/returning behavior.

## Runtime verification
After migration and restart:

```text
get_database_status
view_worker_logs(worker_type="vectorization", log_levels=["ERROR", "WARNING", "CRITICAL"])
semantic_search(project_id=<project>, query=<query>)
```

Expected:
- chunk_count remains valid;
- vectorization continues;
- search results still resolve from chunk to file/project;
- returned chunk/file/project IDs are UUID-shaped where DB identities are exposed;
- no runtime result claims `code_chunks.id` is an integer identity;
- `chunk_uuid` remains a stable business key, not the DB PK.
