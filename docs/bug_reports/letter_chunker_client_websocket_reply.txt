Queue + WebSocket completion (your option 1)
- A way to submit a chunk job and get a job_id (e.g. existing or new queue API).
- A WebSocket endpoint (e.g. /ws) that pushes job completion events (e.g. { "event": "job_complete", "job_id": "...", "result": ... } or a reference to the result).
- Agreed schema for: connection (e.g. path, optional auth), message format, and event types (e.g. job_complete, job_failed, heartbeat).

Bidirectional WebSocket channel (your option 2)
- A WebSocket endpoint that accepts multiple chunk requests on one connection and streams back results (e.g. request id + chunk result).
- Schema for: request format (e.g. { "id": "...", "method": "chunk", "params": { "texts": [...] } }), response format (e.g. { "id": "...", "result": ... }), and optional heartbeat/keepalive.

Planned client API (once server supports it)
We're happy to mirror the embed_client pattern where it fits:
- Queue + WS wait: e.g. submit_chunk_job(texts, ...) returning job_id, and wait_for_job_via_websocket(job_id, timeout=...) that subscribes to /ws (or the agreed endpoint) and returns when the server pushes completion for that job_id. An option like use_websocket=True on a high-level chunk(..., use_websocket=True) could call submit + wait_for_job_via_websocket under the hood.
- Bidirectional channel: e.g. open_bidirectional_ws_channel(receive_timeout=..., heartbeat_interval=...) returning a context manager with send_chunk_request(texts, ...) (or send_json + a small wrapper) and receive_iter() (or equivalent) to consume results as they arrive.

We'll document the required server behaviour (paths, message schema, job_id semantics, event types) in the repo (e.g. docs/chunker_websocket_api.md or similar) so your vectorization worker can align with it. Once the chunker server exposes one or both of these behaviours, we'll add the corresponding client API and keep the existing synchronous chunk() as the default.

We'll follow up with a concrete server schema proposal (draft) and update the client when the server side is ready. If you have constraints or preferences on the /ws schema (e.g. compatible with your existing worker), share them and we'll align.
