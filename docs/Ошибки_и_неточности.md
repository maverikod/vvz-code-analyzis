## Ошибки и неточности, на которые я нарывался в этой сессии

**Параметры MCP-команд:**
- `get_worker_status` без `worker_type` падает — параметр обязательный, хотя по смыслу можно бы дефолтить «все».
- `read_project_text_file` требует `start_line` и `end_line` — нельзя просто передать `file_path`, чтобы прочитать весь файл.
- `view_worker_logs` принимает `log_id` только из жёсткого whitelist (`mcp_server`, `code_analysis`, `vectorization`, `file_watcher`, `indexing_worker`), а не путь к файлу — хотя `list_worker_logs` возвращает именно полные пути.
- `read_only_batch` использует параметр `invocations`, а не `calls` (хотя «calls» интуитивнее).
- `restore_backup_file` требует `file_path` дополнительно к `backup_uuid` — недокументированно.

**Ограничения сервера:**
- `read_only_batch` имеет whitelist разрешённых команд, и `read_project_text_file` в него не входит — параллельное чтение нескольких файлов через batch не работает.
- `read_project_text_file` не умеет читать файлы без расширения (`.gitignore`, `projectid`) — `UNSUPPORTED_FILE_EXTENSION`.
- `restore_backup_file` сломан на бэкенде Postgres: ошибка типа `has_docstring is of type boolean but expression is of type smallint`. Откат бэкапа через MCP **не работает**.
- `fulltext_search` пуст для проекта `mcp_proxy_adapter` — он индексирован метаданными классов, но тексты не векторизованы (0 файлов из 1071), поэтому полнотекстовый поиск ничего не находит.
- `list_project_files` для `mcp_proxy_adapter` падает с `is not in the subpath` — у проекта пути на разных физических локациях.
- `get_code_entity_info` не находит сущность по точечному пути `Config.get` (метод) и даже просто `Config` (класс) — хотя `find_classes` их видит. API `entity_type` не покрывает реальные имена в БД.

**Мои собственные ошибки при правках:**
- При замене строки 26 в `project_overlay.md` положил в `new_lines` исходную строку **плюс** две новые — получил **дубликат** `command_metadata_standard.md` в таблице.
- При второй правке (замена 25-29) забыл воспроизвести **строку с пустой строкой и буллет `Secrets`** — потерял их из секции «Project-specific restrictions». Пришлось третьей правкой восстанавливать.

**Системные/инфраструктурные:**
- File_watcher час крутил петлю с одинаковым WARNING про `projectid below project root`, лог рос со скоростью ~10 МБ/15 минут (5 ротаций за день, 300k+ строк), пока вы не убрали лишние `projectid`-файлы. Воркер не дедуплицирует одинаковые предупреждения между итерациями скана.

**Вывод по работе с этим сервером:** при правках через `write_project_text_lines` нужно **всегда**:
 (1) перечитывать актуальный диапазон **прямо перед** заменой — нумерация смещается после каждой записи; (2) в `new_lines` класть **полное** содержимое заменяемого диапазона, ни одна старая строка из `start_line..end_line` сама не сохраняется. Бэкапы создаются, но откатиться через MCP нельзя — только руками. Поэтому каждая правка должна быть консервативной: сначала read, потом точечная замена в минимальном диапазоне.
