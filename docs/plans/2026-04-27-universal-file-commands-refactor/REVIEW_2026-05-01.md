## 2. Пересечения с планом CST Save Safety

### 2.1. Файлы: косвенное пересечение через общие зависимости

Файлы не пересекаются напрямую. Но универсальные команды делегируют работу в handler’ы, которые вызывают общие компоненты:

- `PythonFileHandler.save/replace/delete` → `run_ops_mode()` → `compose_cst_ops_flow.py` → `apply_replace_ops` → `validate_and_write_temp` → `os.replace`
- `JsonFileHandler.save/replace/delete` → `json_save_tree_to_file()` / `modify_tree()` (свой pipeline)
- `TextFileHandler` → прямая запись + `persist_plain_text_file_metadata`
- `YamlFileHandler` → PyYAML + прямая запись

Универсальные команды реализованы как **MIME-обёртки с маршрутизацией по расширению**. Но Python handler использует `run_ops_mode` (аналог `compose_cst_module`), **не** `save_tree_to_file` (то, что защищает CST save safety).

### 2.2. Два pipeline записи Python-файлов — ключевой пробел
### 2.2. Пути записи: два разных pipeline

| Pipeline | Команды | Функция записи | CST save safety |
|----------|---------|-------------------|------------------|
| Tree-based CST | `cst_load_file` → `cst_modify_tree` → `cst_save_tree` | `save_tree_to_file` | **В объёме** |
| Compose CST | `compose_cst_module`, `universal_file_save/replace/delete` (.py) | `run_ops_mode` → `apply_replace_ops` → `validate_and_write_temp` → `os.replace` | **Вне объёма первой фазы** |

CST save safety README явно указывает: «`query_cst` / `compose_cst_module` (другой путь записи): отдельный подплан».

**Риск:** `PythonFileHandler` использует `run_ops_mode` (compose pipeline), который **не получит** проверки disk snapshot / replay / readback из плана CST save safety. Это не конфликт, но **пробел в покрытии**: universal Python writes останутся без защиты от порчи файла до второй фазы CST save safety.

### 2.3. Рекомендация

Добавить в CST save safety README раздел «Вне объёма первой фазы» явное упоминание `PythonFileHandler` / `universal_file_save|replace|delete` как пользователя compose pipeline, чтобы вторая фаза не забыла об этом пути.
