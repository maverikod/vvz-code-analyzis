It captures the settled architecture for the unified marked-tree model:
node-id markers embedded natively in source content per format, co-located
tree storage, edit-session lifecycle with per-edit git commits, the session
git API exposed via MCP, and consumer integration.

Binding paragraphs below are organized into conceptual blocks mapped to global
steps G-000 and G-001 through G-006. Blocks 1–2 (marker contract and per-format
marking) → G-001. Block 3 (co-located storage and checksum sync) → G-002.
Blocks 4–5 (edit-session lifecycle and session git API) → G-003. Blocks 6 and
8–10 (preview, edit operations, tree query) → G-004. Block 7 (consumer

Block 12 (node identity and map) → G-006. Block 13 is retired; its former
identity-remap integration is absorbed into G-001 through G-006.

Canonical execution order: G-001 through G-005 build and verify the new

after G-001..G-005 are verified. There is no G-007 step. All decisions
described here are settled; open decisions are absent.

## Block 1 — Marker Contract

{a001} A node-id marker is a format-specific token that identifies exactly
one addressable block in the marked content of a tree file. In every format
the in-content marker token is a per-file monotonic integer short_id
(decimal digits, no leading-zero requirement beyond uniqueness). The marker
is a handle, not the canonical identity: it resolves to exactly one UUID4
stored in the tree file's MAP section via the short_id↔UUID map (see
{a002}, {n001}). The short_id never appears outside the tree file and never
crosses the external API boundary.

{a002} A tree file is one file with three ordered sections: (1) CHECKSUMS
first — source content SHA-256 and related checksum fields; (2) MAP second
— short_id↔UUID4 entries, next_free counter, per-node attributes including
block level/type where needed; (3) TREE third — marked content with short_id
markers embedded natively per format. Canonical UUID4 values live only in
the MAP section, never in the TREE section. Markers (short_ids) appear only
in the TREE section. The real source file is never modified and remains
clean. The `unmark` operation extracts and processes only the TREE section
to reproduce original source bytes byte-for-exact.

{a003} The marker contract defines three operations that every per-format
handler must implement: `mark(content) → marked_text`, `parse(marked_text) →
tree`, and `unmark(marked_text) → content`. Markers embedded by `mark` use
integer short_id tokens per {a001}, not UUIDs. The `unmark` operation strips
all marker tokens from the TREE section (and metadata-bound marker forms
within it) and must reproduce the original source bytes exactly — verified
by SHA-256 checksum comparison of `unmark(mark(source))` against `source`.

{a004} The granularity rule: markers address minimally-addressable logical
blocks. Drilldown stops at the leaf addressable block; below it is plain
text. A marker must never split a lexeme, must never be inserted inside a
literal or token, and must never mark a fragment smaller than the
minimally-addressable block for the format.

{a005} Inside an active edit session the tree file is the source of truth.
Outside any session the source file is the source of truth. This two-mode
invariant is absolute and applies to all formats without exception.

{a006} Synchronization between a tree file and the corresponding source file
is done by SHA-256 checksum of source content recorded in the tree file
CHECKSUMS section. If the stored checksum matches the current checksum of
the source file, the tree is valid. If the checksums differ outside an
active edit session, the tree is invalid and must be rebuilt from the clean
source file via TreeBuilder, applying short_id allocation rules in {a007}:
preserve next_free from the existing tree file when one exists; when no
prior tree exists, initialize next_free to 1. Rebuild assigns fresh
short_ids to nodes but validate-and-repair (see {n002}) preserves UUID4 for
nodes whose underlying content identity is unchanged.

{a007} Short_id allocation is per tree file. The MAP section holds
next_free, the lowest integer not yet issued. Each new addressable node
receives short_id = next_free, then next_free increments by 1. A short_id is
never reused after its node is deleted — retired ids remain absent from the
map. On rebuild from source when a prior tree file exists, next_free is
copied from that file's MAP section so new ids continue above all previously
issued ids (including retired ones). On first-time tree creation, next_free
starts at 1. The node-id-map block validate-and-repair operation enforces
map consistency with marked content in the TREE section and keeps UUID4
stable for nodes whose content identity is unchanged across repair.


## Block 2 — Per-Format Marking

{b000} Every format has a dedicated handler that implements a common
four-operation contract: `parse_content(file_path, content) → list[TreeNode]`
builds the node tree from source content; `mark(content) → marked_text`
embeds node-id markers natively into the content using the format's own
syntax; `unmark(marked_text) → content` strips all markers and reproduces
the original source bytes exactly; `sidecar_path(source_abs) → Path` returns
the sibling tree-file path for the given source file. All handlers are
registered in a central registry keyed by file-extension and live under
`core/tree_handlers/`. No handler may use another format's marker syntax.
The common contract is the only coupling between handlers; each handler's
internal implementation is entirely format-specific. Every addressable node
receives a short_id marker in the TREE section per format rules in
{b001}–{b005}. Canonical UUID4 for each node is written only in the MAP
section, not by handlers inventing API-facing ids. Handlers use the
node-id-map block ({n001}–{n003}) to resolve short_id↔UUID when building
or parsing trees; they do not embed UUIDs in markers.

{b001} Text files (`.txt`, `.rst`) use line-level markers. Each marked line
in the TREE section is prefixed with `<short_id>:` followed by the original
line content, where short_id is a decimal integer. Paragraph vs continuation
line level is recorded in MAP attributes for the node, not encoded in the
marker prefix. Paragraph boundaries are the only structural delimiter: two or
more consecutive newlines (`\n\n`) mark a paragraph break; single newlines
are continuation. The handler produces exactly two node levels — paragraph
and line — and no deeper nesting. Blank lines that serve as paragraph
separators carry no marker and are reproduced verbatim by `unmark`. Prefix
width for preview alignment is sized for integer short_id magnitude, not
UUID width (see {i007}).

{b002} Markdown files use HTML comments as markers, placed immediately before
each addressable block. The comment form is `<!-- id:<short_id> -->` on its
own line, where short_id is a decimal integer. Parsing uses `markdown-it-py`
(4.0.0, already installed in venv; `markdown`, `mistune`, `commonmark`, and
`marko` are not installed and must not be used). Addressable blocks are: ATX
headings (`#`..`######`), paragraphs, fenced and indented code blocks,
blockquotes, and list items. A heading marker precedes the heading line; all
other markers precede the block's first line. Inline spans (bold, italic,
links) within a paragraph are not addressable and carry no marker. The
`unmark` operation removes all `<!-- id:... -->` comment lines and must not
alter any surrounding whitespace or blank lines.

{b003} YAML files use an `___id___` key as the first entry of each
addressable mapping block. The value of `___id___` is integer short_id.
Sequence elements that are scalars are wrapped into a single-key mapping
`{___id___: <short_id>, v: <scalar>}` to carry a marker. The marked file
must remain valid YAML parseable by standard libraries; the `___id___` key
is the only addition. The `unmark` operation removes all `___id___` keys
(and the `v`-unwrap for scalar sequence elements) and must reproduce the
original YAML byte-for-byte. Multiline scalar values (literal `|` and folded
`>` blocks) are one addressable unit and their internal newlines are not
split.

{b004} JSON files use a recursive id-wrapper. Every JSON value at every
nesting level — object, array element, or root scalar — is replaced by a
wrapper object `{"___id___": <short_id>, "v": <original_value>}`, where
short_id is a JSON number (integer). The wrapped file must remain valid
JSON. The `unmark` operation extracts the `v` field at every level
recursively and reconstructs the original JSON structure exactly, including
all whitespace and key ordering within objects. The wrapper is applied
uniformly: even a top-level string `"hello"` becomes
`{"___id___": 1, "v": "hello"}`. The concrete wrapper shape is fixed at
this design; no alternative shapes are permitted.

{b005} Python files use a hybrid scheme determined by CST node type. Where a
CST node carries a metadata dict (function definitions, class definitions,
decorated statements, and similar named constructs that already store
metadata in the existing codebase), the short_id is written into that
metadata dict under a reserved key. Where a CST node has no metadata dict
(simple statements, expressions, import lines, assignment targets), a
trailing comment `# ___id___:<short_id>` is appended at the end of the
last physical line of the logical block. No per-node UUID appears in marked
content or CST metadata — canonical UUID4 lives only in the MAP section.
The minimally-addressable block is the complete logical statement or
expression as parsed by the CST, never an individual physical line within
it. A multiline docstring is one block; its marker appears on the line
containing the closing `"""`, after the quotes. Line continuations (`\`)
and bracket-continuation expressions (open paren/bracket across lines) are
one block; the marker appears on the last physical line of the continuation.
A shebang line (`#!/...`) is not addressable and carries no marker. The
`unmark` operation removes metadata-dict short_id entries and trailing
comments without altering any other whitespace, and must reproduce the
original source byte-for-byte.


## Block 3 — Co-Located Storage and Checksum Sync

{c001} Tree files are stored as sibling files directly next to the source
file they describe. A tree file named `<stem>.<ext>.tree` lives in the same
directory as `<stem>.<ext>`. All formats use this single convention; no
subdirectory is introduced. Placing the tree file next to the source makes
it immediately visible in a file listing: if `foo.py.tree` is present, the
tree exists; if absent, it does not. G-002 (Tree Storage and Lifecycle)
establishes this co-located sibling storage convention as the canonical
on-disk placement rule, superseding the central `.trees/` mirror directory,
the per-Python `.cst/` subdirectory, and the adjacent `.tree_sidecar` layout
patterns. The physical removal of those legacy layouts and cutover of all
resolvers to the sibling convention is owned by G-000 (SidecarLayoutMigration).

{c002} The `resolve_sha_sync_policy` function (already implemented in
`code_analysis/commands/universal_file_edit/sha_sync_policy.py`) governs the
four-branch decision: no sidecar present, checksum match, checksum mismatch
with no active session, checksum mismatch with an active session. This policy
is applied uniformly to all formats.

{c003} On checksum mismatch outside an active session, the tree is rebuilt
from the source file using the appropriate per-format handler. This rebuild
writes a new tree file co-located next to the source and records the new
source checksum in the tree file.

{c004} A dedicated tree-validity module (`core/tree_lifecycle/checksum.py`
or equivalent) exposes two pure functions: `compute_content_checksum(content:
str) → str` computes the SHA-256 hex digest of source content;
`is_tree_valid(content_checksum: str, stored_checksum: str | None) → bool`
returns True if and only if both values are non-None and equal. These
functions have no file-system side effects and no format-specific logic.

{c005} A dedicated tree-creation module (`core/tree_lifecycle/builder.py`
or equivalent) exposes `TreeBuilder.build(*, content, source_abs, file_path,
content_checksum)`. It resolves the appropriate FormatHandler via
`HandlerRegistry.default_registry().resolve(source_abs)`, produces
natively-marked tree content via `FormatHandler.mark(content)`, writes the
tree file to `FormatHandler.sidecar_path(source_abs)` as a sibling of the
source file, and records `content_checksum` in the tree file AND exposes it
in the returned tree reference so that subsequent validity checks can compare
it against the current source checksum. This module is the single place that
writes tree files on the canonical unified path; no other module writes tree
files directly. The legacy `recreate_tree_from_content(*, kind, ...)` dispatch
remains in the codebase as an untouched parity oracle until G-000 cutover.

{c006} `core/tree_lifecycle/` is the single owner of tree validity and tree
creation. Every consumer — preview, grep, watcher, indexer, edit sessions,
builders — routes through this module. No ad-hoc per-call-site tree
validation or tree creation exists outside it. A thin file-path wrapper
(`validate_or_recreate_tree_file(project_root, file_path, force=False)`) reads
the source file once, computes its checksum, and delegates to the validity
and creation functions; this wrapper preserves backward compatibility for
call sites that work with paths rather than content.


## Block 4 — Edit Session Lifecycle

{d001} An edit session is represented by a session directory placed next to
the source file. The directory name is `<FileName>-<UUID4>` where `FileName`
is the source file's base name and `UUID4` is a randomly generated UUID.

{d002} Opening a session performs four steps in order: create the session
directory; copy the source file into the session directory; validate the
co-located tree file by checksum — if valid, copy it into the session
directory, if missing or invalid, build a new tree from the source content
inside the session directory; initialize a git repository (dulwich 1.2.4,
already installed) inside the session directory and create the initial commit
containing the source copy and the tree file. When the caller supplies
`content` for open: that content is accepted ONLY when the target file is
NON-EXISTENT or its source/tree is BROKEN/CORRUPTED (invalid checksum,
unreadable tree, or missing co-located tree where required). If the target
file exists and is valid, supplying `content` is an ERROR. When content is
supplied for a missing or broken file: FIRST write the provided source into
the session code file, THEN follow the standard open protocol — build the
tree, write the tree, and record checksums (КС) — and create the normal
session git initial commit. The session records whether the in-session tree
is valid (source parses and checksum matches) or invalid (source cannot
parse).

{d003} While the in-session tree is valid (per {h008}), all editing operates
on the tree file inside the session directory; the tree is the source of
truth. Before each mutation, any short_id markers in the TREE section are
hidden into the tree content as format-native comments (or equivalent inline
marker form); the mutation is applied to that temporarily denuded
representation; after the mutation, short_id markers are fully restored to
their canonical positions — the short_id lives in metadata where the format
provides a metadata slot, and as a format-native comment at the END of the
block where it does not. The CHECKSUMS and MAP sections are preserved across
mutations; UUID4 values in MAP are not altered by hide/unhide. Every tree
modification without exception must be followed immediately by: writing the
updated tree file to disk inside the session directory; exporting the
in-session source copy by stripping all special marker comments from the
TREE section via `unmark` and writing that clean source into the session
directory (the session TEMP artefacts mirror the original project's source
and tree layout, not the live project files); recording SHA-256 content
checksums for both the tree file and the exported source copy; and creating
a git commit in the session repository that captures both artefacts. One tree
modification equals one git commit. When the in-session tree is invalid (per
{h008}), the per-mutation cycle is DEGRADED: the TREE section holds plain
source with no short_id markers; the CHECKSUMS and MAP sections are preserved
unchanged; write only the updated in-session source copy to disk inside the
session directory, record its SHA-256 content checksum, and create a git
commit in the session repository capturing that source artefact only — no
marker hide/unhide and no tree-file write; one plain-text modification
equals one git commit. When the tree becomes valid again (per {h009}), the
full invariant above applies without exception.

{d004} The write command is two-stage at the API boundary and gates only the
external copy-out; in-session artefacts are already finalized per mutation by
{d003}. Stage 1 (preview): the command computes the diff between the
in-session artefacts and their external counterparts without modifying any
external file. The in-session side is the unmark-exported source copy and the
in-session tree file held inside the session directory. The external side is
the live project source file and its co-located sibling tree file. The
command returns both diffs. If there are no diffs — the in-session source
matches the live source and the in-session tree matches the external tree —
the write is a no-op and nothing is written externally. This write-stage
external diff is distinct from `session_git_diff` ({e003}): it compares
in-session artefacts against live external files, not against session git
history.

{n7k2} Stage 2 (confirm): only upon an explicit confirmation — a second
call or a `confirm` parameter on the write command — does the command
atomically copy both artefacts out to their external co-located positions:
the in-session unmark-exported source over the live project file, and the
in-session tree onto its sibling tree path next to that source, replacing
the previous external copies. Atomic means both are written externally or
neither is. Without explicit confirmation, no external file is modified.

{d005} Closing a session deletes the session directory entirely, including
its git repository, without performing any validation, checksum, or
consistency checks. Only the final co-located tree file survives (if
previously written out via the write command). The git history of the session
is ephemeral and does not persist after close.

{d006} The dependency on `dulwich` must be declared in the project's
dependency manifest (requirements file or pyproject.toml). It is currently
installed in the venv but not declared.
## Block 5 — Session Git API

{e001} The session git repository is exposed through MCP as a set of
session-scoped commands. Every command in this set requires a `session_id`
parameter. A command called without an active session returns an error; it
does not attempt to access the file system without a valid session context.

{e002} `session_git_log(session_id)` returns the commit history of the
session repository: commit hash, message, and timestamp for each commit.
Implementation reads history via `repo.get_walker()` (dulwich); `porcelain.log`
is not used because it writes to stdout and returns None.

{e003} `session_git_diff(session_id, *, mode, rev_a, rev_b=None)` returns a
diff in one of two modes. Mode `tree` diffs two tree-file versions (two
commits) and shows what changed at the marker/node level between them. Mode
`source` diffs a tree-file version (rev_a) against the source file held inside
the session directory, showing divergence between the marked tree and that
in-session source. The in-session source is the copy made at open and re-exported
after every mutation via `unmark` (per `{d003}`); the live project file on disk
is not an input to this diff. Mode `source` therefore compares `rev_a` against
the most recent per-mutation exported in-session source.

{e004} `session_git_show(session_id, rev)` returns the full content of the
tree file at the given commit.

{e005} `session_git_status(session_id)` returns uncommitted changes relative
to HEAD. Under the invariant that every tree modification immediately
produces a commit, this should always be empty; the command exists as a
consistency check.

{e006} `session_git_revert(session_id, rev)` rolls the session tree back to
the state at `rev` by creating a new revert commit. The git history is
preserved; no commit is deleted. Because one tree modification equals one
commit, revert rolls back by individual node edits.
## Block 6 — Preview Over All Formats

{f001} The `universal_file_preview` command must build a valid marked tree
for any file format before returning navigation results or node content. If
no co-located tree exists or the existing tree is invalid by checksum, the
command builds a new tree via the `core/tree_lifecycle/` module before
proceeding. Python files use the CST builder; all other formats use their
respective marked-tree handlers.

{f002} Node references returned by preview are short_id integers valid for
the requested file's tree. These ids are stable within a tree version and
are used to address subsequent edits, diffs, and lookups. UUID4 is never
returned at the API boundary.


## Block 7 — Consumer Integration

{g001} The grep subsystem (`fs_grep_structural_integration.py`,
`grep_block_resolver.py`, `fs_grep_command.py`) routes all tree-validity
checks through `core/tree_lifecycle/`. On a checksum mismatch it rebuilds
the tree before binding a grep match to a node id. A rebuild failure yields
an ineligible match (lower relevance), not a structural finding.

{g002} The file watcher (`core/file_watcher_pkg/processor_queue.py`) calls
the `core/tree_lifecycle/` file wrapper before inserting or updating a file
row in the database. The computed source checksum is persisted in the `files`
table as a `tree_checksum` column (added via idempotent `ALTER TABLE`). No
migration is required; the column is added on first startup if absent.

{g003} The indexer (`core/database/files/update.py`) uses a checksum-based
skip guard: if the current source checksum equals `files.tree_checksum`, the
file is not reindexed; if they differ, reindex proceeds. Reindexing is
content-driven.

{g004} Search-session findings (`Finding`) do not carry file mtime and ranking
does not use mtime as a sort dimension; ordering uses relevance score with a
deterministic `result_id` tie-break.


## Block 8 — Tree Edit Operations

{h001} Every per-format handler exposes a uniform set of six tree-edit
operations addressed by short_id: insert, delete, replace, move,
edit-attributes, and edit-content. All six operations are available for all
formats; the handler translates each operation into the format-appropriate
manipulation of the marked tree file. No operation bypasses the marker
addressing system; every target is identified by its short_id, not by line
number or byte offset.

{h002} Insert adds a new addressable block at a position relative to an
existing short_id: before, after, or as first/last child where the format
supports nesting. The inserted block receives a fresh short_id assigned per
{a007}. The handler validates that the inserted content is syntactically
valid for the format at the insertion point before writing.

{h003} Delete removes the block identified by the given short_id from the
tree, together with its subtree if the block has children. For formats that
require structural completeness (JSON, YAML), the handler ensures the
resulting tree remains valid after deletion — for example, removing the only
element of a required object key is rejected, not silently mangled.

{h004} Replace substitutes the content of the block identified by a short_id
with new content while preserving the short_id. The replacement content must
be syntactically valid for the format at that position. The short_id is
retained so that external references to the block remain valid after the
replacement.

{h005} Move relocates a block (identified by its short_id) to a new position
within the same tree, addressed relative to another short_id. The block's
short_id is preserved. Move is semantically equivalent to delete-then-insert
but executes atomically within the handler so that no intermediate invalid
state is written to the tree file.

{h006} Edit-attributes modifies the metadata attributes of a node without
changing its content or position. For Python CST nodes, this updates the
metadata dict entries. For other formats that carry per-block attributes
(e.g. YAML anchors, JSON schema annotations if present), the handler updates
the attribute in place. The short_id is not changed.

{h007} Edit-content modifies the text content of a leaf block (one that has
no addressable children) without changing its short_id or position. For
non-leaf blocks the handler rejects the operation; the caller must drill down
to a leaf node first. The updated content must be syntactically valid for the
format at that leaf position.

{h008} After a session opens, the in-session tree is classified as VALID when
the source content parses and its checksum matches the stored checksum, or
INVALID when the source content cannot parse (for example, broken Python
indentation). While the tree is VALID, all edits use short_id addressing and
the six tree-edit operations of {h001} through {h007}; each mutation follows
the full per-mutation invariant of {d003} including short_id hide/unhide and
tree write. While the tree is INVALID, edits are plain-text only: callers
supply string or line-range replacements applied directly to the in-session
source copy; short_id addressing, marker hide/unhide, and tree-file writes are
forbidden; the TREE section holds plain source with no short_id markers while
the CHECKSUMS and MAP sections are preserved. Each plain-text mutation
follows the DEGRADED per-mutation invariant of {d003}: write the updated
in-session source, record its SHA-256 checksum, and create a session git
commit — nothing else.

{h009} Re-validation: when a plain-text edit during an INVALID session
produces source that parses successfully, the session rebuilds the tree via
unified `TreeBuilder.build` from the updated source content, then restores
short_id markers in the TREE section via the node-id-map block build
operation ({n002}) while preserving UUID4 entries and next_free from the
existing MAP section; records checksums; and switches back to VALID mode.
Short_id addressing and the full {d003} per-mutation invariant resume from
that point forward.


## Block 9 — Preview Navigation and Node Rendering

{i001} Preview navigation follows a three-phase procedure for every request.
Phase 1 (enumerate): open the tree for the file, resolve the focus node from
the given `node_ref` (or use the tree root if `node_ref` is absent), and
produce the focus node's ordered list of direct child blocks. Phase 2
(select): apply a selector expression to the block list to choose a subset.
Phase 3 (render): for each selected block produce a rendered representation
including its node id, type label, attributes summary, and a text excerpt.
This procedure is identical for all formats and both in-session and
out-of-session access.

{i002} A `node_ref` in a preview request is always a short_id (integer)
valid for the requested file's tree — never a line number and never a
UUID4. The server resolves short_id to internal UUID4 via the MAP section
of the loaded tree file. An unrecognised short_id returns an input error; it
is never silently ignored.

{i003} When a `session_id` is supplied in the preview request, navigation
operates on the in-session tree or draft held by the active `EditSession`
(in-memory CST for Python, `{file}.draft` and in-memory roots for JSON/YAML,
`{file}.draft` for text) rather than only the last committed on-disk
artefacts. This is the mechanism for previewing unsaved edits before write.
A `session_id` referencing a closed or non-existent session returns an error.

{i004} For JSON and YAML files during an active edit session, a `node_ref`
that is a short_id (integer) locates the node in the tree via the MAP
section, walks up to the nearest ancestor whose discriminator is `object` or
`array`, and exposes that ancestor's children as the block list. A
`node_ref` that is an `object` or `array` node drills directly into that
node's children. A scalar `node_ref` (string, number, boolean, null leaf)
triggers the walk-up policy. An unrecognised short_id returns an input error.

{i005} A selector expression applied in phase 2 may be a slice string of the
form `start:end` (zero-based, exclusive end) to cap the returned block count,
or a list of specific node ids to cherry-pick blocks. Full inline rendering of
a node's source text (annotated mode with node-id prefixes at the start of
each physical line) is used when the node's line span is strictly less than
that format's `full_text_max_lines` threshold; when the span equals or
exceeds the format's threshold, or when that format's threshold is zero,
preview falls back to collapsed structural drilldown. Each supported format —
Python (`.py`), JSON (`.json`), YAML (`.yaml`/`.yml`), Markdown (`.md`), and
plain text (`.txt`/`.rst`) — has its own independent `full_text_max_lines`
value; the default is 200 source lines for every format unless configured
otherwise. Setting a format's threshold to zero disables full inline
rendering for that format (drilldown only). Thresholds are measured in source
lines for every format, not bytes. For JSON specifically: content must be
normalized to a printable (pretty-printed) form BEFORE line counting and
inline rendering, so the line-based threshold is meaningful on the JSON
preview/handler path. A separate `max_chars` budget may paginate the serialized
preview envelope but does not control the inline-versus-drilldown decision.

{i006} Preview for Python files produces CST-node blocks with type labels
(e.g. `FunctionDef`, `ClassDef`, `ImportFrom`), source line ranges
(`L<start>-<end>`), and attribute summaries. In block enumeration and
collapsed text rendering, `IndentedBlock` wrapper nodes are not emitted as
separate blocks; their direct statement children appear in place of the block
container. When rendering inside compound statements (`If`, `For`, `While`,
`Try`, `With`, `Match`, function/class bodies), the compound statement header
line may still appear as a summary row with an ellipsis when nested compound
statements are present; drilldown follows CST parent-child relationships via
stable ids in sidecar metadata. A `tree_id` from a prior `cst_load_file` call
may be passed instead of `file_path` to reuse an already-loaded CST.

{i007} In preview annotated and full-text rendering modes, every line that
has an associated short_id is prefixed at the start of the physical source
line with `[<short_id>] ` (decimal integer) followed by the original line
content. Lines without an associated short_id use blank padding so column
alignment of source text is preserved across the rendered block; the padding
width is the minimal width needed for the maximum short_id digit count in
the tree (integer string length), not a fixed cap and not UUID width.

{i008} Before preview renders Python source text, legacy identity comments
are removed from the logical source used for display: the trailing
`# cst-node-ids: begin` … `# cst-node-ids: end` block and whole-line
`# @node-id: <uuid>` comments. Stripping occurs on both committed on-disk
source and in-session draft lines used for diff preview. Preview output never
shows these special comments; node ids in preview come from short_id prefixes
derived from TREE section markers and the MAP section, not from preserved
inline comment markers or UUID values.


## Block 10 — Tree Query (XPath-Compatible Search)

{j001} The tree query system provides structural search over the marked tree
of any file, in both preview mode and edit-session mode. It is not a
full-text or database search; it operates on the in-memory or on-disk tree
structure and returns short_id integers of matching nodes. The result
short_ids are directly usable in subsequent preview navigation and tree-edit
operations.

{j002} Two search modes are supported. Simple mode filters nodes by
combinations of: node type label, node name, qualified name (for Python
class/method hierarchy), and line range (start_line, end_line). Simple mode
is available for Python CST trees and is not format-universal. XPath mode
uses a CSTQuery selector string and is the primary format-agnostic mode.

{j003} The CSTQuery selector syntax is XPath-compatible and supports: `//`
for recursive descendant traversal (e.g. `//FunctionDef` finds all function
definitions at any depth); `@attr` for attribute access in filter predicates
(e.g. `[@name='foo']`); numeric comparison operators `>=`, `<=`, `>`, `<`
on numeric node attributes including `start_line`, `end_line`, and
`children_count`; and the `:not(selector)` pseudo-class for negation. A full
selector example: `//ClassDef//FunctionDef[@name='execute'][start_line>=100]`.

{j004} When `require_one` is set in a query, the system requires exactly one
matching node; if zero or multiple nodes match, the response carries a
structured error (`NoMatch` or `NonUniqueMatch`) with the candidate list.
This mode is used when the caller needs a unique anchor for a subsequent edit
operation and must not silently proceed on an ambiguous match.

{j005} When `include_code` is set, each matching node's result includes the
source code of that node (extracted via the CST's `code_for_node` mechanism),
eliminating the need for a separate preview round-trip to inspect the matched
node's content. This option is only meaningful for Python CST trees.

{j006} In an active edit session the query runs against the session's
in-memory tree draft, not the disk tree file, so it reflects unsaved edits.
Outside a session the query runs against the validated on-disk tree (rebuilt
from source if the checksum is stale). The same CSTQuery selector syntax
applies in both modes.
## Block 11 — Legacy Sidecar Layout Cutover

{k001} This block (G-000) is the final sidecar layout cutover. It runs after
the new marked-tree system (Blocks 1 through 10, global steps G-001 through
G-005) has been built and verified end to end. Node identity is handled by
G-006 NodeIdMap in the tree-file MAP section; this block is the final layout
cutover with no subsequent step. The new system introduces the sibling convention of {c001} and the `sibling_tree_path`
function that implements it. Several existing call sites in the codebase are
not reached by the consumer-integration work of Block 7 and therefore still
resolve tree-file paths through the three legacy layouts: the per-Python
`.cst/` subdirectory (path form `<parent>/.cst/<stem>.tree`); the central
`.trees/` mirror directory under the project root (path form
`<project_root>/.trees/<rel_path>.tree`); and the adjacent `.tree_sidecar`
files (path form `<source_abs>.tree_sidecar`, used for text and markdown).
This block redirects those remaining call sites onto `sibling_tree_path` and
then removes the legacy layouts entirely. It does not create
`sibling_tree_path`; that function already exists from the storage block.

{k002} The three legacy layouts are produced by three independent path
resolvers, none of which is a delegate of another. Each is replaced by
`sibling_tree_path`, and the surrounding dead code is deleted. (1) The Python
resolver `sidecar_path_for_py` in `cst_tree/tree_sidecar.py` produces the
`.cst/` path; it is removed, and `write_sidecar_atomic` (same module) is
redirected to write at the sibling path. Its direct callers
(`tree_builder.py`, `tree_saver.py`, `structure_extraction/stable_tree.py`,
`grep_block_resolver.py`, and the universal-file-edit commands open/close,
`invalid_write_support`, `sidecar_cst_apply`) follow the sibling path through
these changes. (2) The tree-temp resolver `resolve_trees_sidecar_path` in the
standalone module `core/tree_temp/sidecar_paths.py` produces the `.trees/`
path; the whole module (including `TREES_DIRECTORY_NAME` and `SidecarPathError`)
is removed, and its direct callers (`tree_temp_write_commit.py`,
`tree_temp_open_support.py`) are pointed at `sibling_tree_path`, so JSON and
YAML tree files become siblings of their sources like every other format.
(3) The text/markdown `.tree_sidecar` path, computed inline inside the
format-dispatch resolver `sidecar_path_for` in
`search_session/tree_representation.py`, is removed; `sidecar_path_for`
resolves to `sibling_tree_path` for every format, and the reader functions in
the same module (`_read_sidecar_state`, `_read_cst_digest_and_root`,
`_read_tree_temp_digest`, `_read_adjacent_sidecar_digest`, and related) read
from the sibling path. The `TreeRepresentationRef` produced by
`validate_or_recreate_tree_file` carries the sibling path, not a legacy path.

{k003} Existing tests and scripts that assert or construct `.cst/<stem>.tree`,
`.trees/<rel>.tree`, or `<source>.tree_sidecar` paths are updated to the
sibling path instead. No backward-compatibility shim for legacy paths is
introduced; the cutover is hard. After the cutover, a file-system scan for
`.cst/*.tree`, `.trees/*.tree`, and `*.tree_sidecar` artifacts inside the
project must return zero results.

{k004} The legacy pending-sidecar mechanism is removed, not migrated. The
functions `pending_sidecar_path_for_py` (path form
`<parent>/.cst/_pending/<stem>.tree`), `resolve_sidecar_write_path`,
`promote_pending_sidecar_to_final`, and `_cleanup_empty_pending_dir` in
`cst_tree/tree_sidecar.py`, together with the `_pending` subdirectory and the
`is_file()` branch inside `write_sidecar_atomic` that chooses between the
final and pending paths, are all deleted. The pending mechanism existed only
to support the pre-session write path, where a tree could be written while
its source file did not yet exist on disk. Under the edit-session lifecycle
of Block 4, this state no longer occurs externally: a create-flag open
materialises the empty source and its tree only inside the session directory
({d002}), and external artifacts appear only at the first session write,
where the regenerated source and its sibling tree are copied out together and
atomically ({d004}). There is therefore never an external tree without an
external source beside it, and `write_sidecar_atomic` after the cutover always
writes at the single sibling path with no pending branch.

## Block 12 — Node Identity and Map

{n001} Canonical node identity is UUID4. Each addressable node has exactly one
UUID4 stored in the tree file MAP section. UUID4 is never embedded in marked
content, never returned by preview/edit/query/grep MCP commands, and never
accepted as an external node reference. UUID4 is assigned at node creation
and kept stable across validate-and-repair for nodes whose underlying content
identity is unchanged.

{n002} The node-id-map block is a dedicated module exposing exactly three
operations: (1) build — scan the tree's TREE section and MAP section to
generate or refresh the short_id↔UUID map and per-node UUID entries; (2)
validate_and_repair — detect inconsistencies between markers, map, UUID
entries, and next_free, then fix them while preserving UUID4 for unchanged
nodes per {a007}; (3) resolve — bidirectional lookup short_id→UUID and
UUID→short_id within one tree file context. No other public entry points
perform map mutation. This block replaces the former universal-node-id
generate/validate pair and centralizes map logic previously split across
format handlers.

{n003} External consumers (preview navigation, edit operations, tree query,
grep/indexer block assembly) address nodes by (project-relative file path,
short_id) only. short_id is the integer marker token from the tree file for
that file. Resolution to UUID4 happens only inside the server when operating
on a loaded tree, via the node-id-map block. An unrecognised short_id for
that file is an input error, never silent ignore. File path disambiguates
short_id — the same integer may exist in different files' trees
independently.
