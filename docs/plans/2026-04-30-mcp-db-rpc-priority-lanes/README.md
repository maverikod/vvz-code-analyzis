## Целевая схема слоёв (команда → СУБД)

Унифицированная цепочка в продукте:

```text
MCP-команда / воркер
    → DatabaseClient (универсальный фасад)
        → rpc_client.call()   ← без очередей и без искусственной задержки: сразу вниз
            → [транспорт in-process]
                → process_rpc_request()  ← только маршрутизация по method
                    → RPCHandlers
                        → PostgreSQLDriver / SQLiteDriver  ← здесь пулы, блокировки на conn, политика
                            → СУБД (5 conn для PG: 3 write + 2 read — см. ниже)
```

- **Универсальный слой** (`DatabaseClient`, `InProcessRpcClient` в части «просто вызвать»): **сыплет** запросы вниз к обработчику **без** собственных очередей и **без** добавляемых задержек. Не реализует пул, не считает «первое свободное соединение».
- **Специфический драйвер (PostgreSQL, фаза 1):** держит **ровно пять** соединений с БД; запрос уходит **в первое свободное** соединение из нужного пула: **три** — на **запись**, **два** — на **чтение**. Классификация операции (read vs write) — в драйвере / слое операций (`PostgreSQLDriver`, `PostgreSQLOperations`, менеджер транзакций), не в универсальном клиенте.
- **Транспорт in-process сегодня:** один `threading.Lock` на весь `call` **сериализует** всё до драйвера — это нужно **снять или сузить** после того, как драйвер стал безопасен при конкуренции на пуле (шаги 2–2a).
- **SQLite (subprocess):** **фаза 2**; см. [фаза 2](#phase-2-sqlite). Там **нельзя** копировать «3+2 писателя» к одному файлу БД: у SQLite другая модель (один writer к журналу); политику соединений задаёт **специфический** SQLite-путь отдельно.

## Парадигма (нельзя нарушать)

| Правильно | Неправильно |
|-----------|-------------|
| Пул **5** соединений (**3** write, **2** read) для self-managed execute, выдача **первого свободного** в нужном пуле — в **`PostgreSQLDriver`** (и связанных модулях операций/транзакций), с потокобезопасностью; основное `conn` и транзакции — вне этого пула | Пул, балансировка и «свободное соединение» в `DatabaseClient` / универсальном RPC-клиенте |
| Универсальный путь **сразу** вызывает драйверный обработчик, **без** очередей и намеренных задержек на этом слое | Очередь «интерактив / фон» в универсальном слое вместо конкуренции на пуле в драйвере (если только это не временный мост до готовности пула — явно пометить как временное) |
| Классификация read vs write для выбора пула — в специфическом драйвере / операциях | Логика выбора пула в **`rpc_dispatch.py`** |
| Для SQLite — **своя** модель в subprocess (`RPCServer` / `RequestQueue`), не копировать 3+2 writer к одному файлу | Три одновременных writer-соединения к одной SQLite БД «как у Postgres» |

**Соответствие текущему коду:** нарушений парадигмы сейчас нет; `_priority_for_request` — рантайм SQLite. Для Postgres целевое состояние — **пул в драйвере**, а не «умный» универсальный клиент.

## Проблема

Сервер зависает под нагрузкой индексирования. Для **PostgreSQL** все обращения к БД из MCP и воркеров идут через **in-process** путь: один **`threading.Lock`** в `InProcessRpcClient` — **нет очередей и нет приоритетов**, потоки выстраиваются в FIFO на `acquire()`. Длительный захват лока воркером блокирует `list_projects` до таймаута (~15s).

**Важно:** инфраструктура `RequestQueue` / `RequestPriority` в `rpc_server.py` относится к **SQLite subprocess** и **не используется** в прод-пути PostgreSQL. См. отчёт, раздел «Два рантайм-пути».

**Наблюдаемые симптомы** (как в исходном ТЗ):

- `list_projects` → `LIST_PROJECTS_TIMEOUT` после 15s
- `indexing_worker` может при этом показывать нормальное время на файл
- `vectorization_worker`: тяжёлые операции удерживают тот же lock
- `get_worker_status` может отвечать без того же узкого DB-пути

**Логическая схема узкого места (PostgreSQL):**

```text
MCP-команда (list_projects)
    └──> DatabaseClient → InProcessRpcClient.call()
              └── threading.Lock → process_rpc_request → PostgreSQLDriver
                                                    ^
         indexing_worker (index_file) ─────────────┘
         vectorization_worker (execute / execute_batch)
```

## Цель

MCP-команды **не должны** блокироваться длительной фоновой активностью воркеров на **PostgreSQL in-process** канале.

**Жёсткое требование:** `list_projects`, `get_database_status` и read-only команды через этот канал укладываются в таймаут при полной нагрузке воркеров (в разумных пределах нагрузки СУБД).

---

## Решение (PostgreSQL in-process): пул соединений в специфическом драйвере

**Не создавать** второй DB-процесс. **Не** строить основную конкуренцию на очередях в универсальном клиенте.

**Канон (фаза 1):**

| Параметр | Значение |
|----------|----------|
| Число соединений в **пуле** self-managed | **5** (отдельно: одно основное `conn` драйвера и коннекты менеджера транзакций — см. [статус аудита](#статус-после-аудита-corrections-2026-05-01)) |
| Запись | **3** соединения в пуле; запрос, требующий write-пула, уходит на **первое свободное** из этих трёх |
| Чтение | **2** соединения в пуле; read-путь — **первое свободное** из двух |
| Универсальный слой | Проброс вызова **без** очередей и **без** добавляемых задержек |
| Где код | `PostgreSQLDriver`, менеджеры транзакций, `PostgreSQLOperations` — по файлам и «1 файл = 1 шаг» |

**Транзакции:** одна транзакция **закрепляет** одно соединение на время `begin`…`commit`/`rollback` (как в типичном пуле).

**Опционально (в дополнение к пулу):** поле **`priority`** в `RPCRequest` и теги воркеров (шаги 1, 3–4) — для подсказки маршрутизации или метрик, **не** как замена пулу. Если после пула таймауты MCP исчезают, часть шагов можно упростить по результатам замеров.

**Ожидание при полном пуле:** блокировка на получении conn — **внутри драйвера** (per-pool mutex + condition / очередь ожидателей **только на conn**), не в `DatabaseClient`.

**Starvation / fairness:** при полном write-пуле длительными воркерами read-пул всё равно может обслуживать MCP — это главный эффект схемы 3+2. Дополнительные лимиты (например max wait) — по необходимости в драйвере.

**Устаревший вариант «две lanes + один conn»** из ранних черновиков плана — **не** целевой; при необходимости оставить только как временный мост в ADR до готовности пула.

## Фаза 1: PostgreSQL — план реализации («1 файл = 1 шаг» при дроблении)

### Шаг 1 — поле `priority` в `RPCRequest`

- **Файл:** `code_analysis/core/database_client/protocol/rpc_protocol.py`.
- Добавить **`priority: int = 0`** в dataclass. **Порядок полей:** после `params`, перед `request_id` (у `dataclass` порядок важен для позиционных аргументов и единообразия с остальным кодом).
- **`to_dict`:** уже есть идиома `if self.request_id is not None: result["id"] = ...` — для `priority` использовать ту же схему опционального ключа: **`if self.priority != 0: result["priority"] = self.priority`** (дефолт 0 не раздувает JSON).
- **`from_dict`:** уже читает опциональные поля через `data.get(...)` — добавить **`priority = int(data.get("priority", 0))`** и передать в конструктор.
- **Тесты (минимум):** unit на round-trip `to_dict` / `from_dict` с ненулевым `priority` и с отсутствием ключа в dict (ожидается 0).
- **Совместимость SQLite subprocess:** сервер может игнорировать поле до фазы 2; если какой-то парсер ломается на лишнем ключе — отдельный микрошаг **только при** падении тестов (не планировать преждевременно).
- **Примечание:** in-process PostgreSQL передаёт `RPCRequest` как объект; дефолт `priority=0` сохраняет обратную совместимость.

### Шаг 2 — пул из пяти соединений (3 write + 2 read) в специфическом драйвере PostgreSQL

- **Новый модуль (рекомендуется):** `code_analysis/core/database_driver_pkg/drivers/postgres_connection_pool.py` — класс пула (выдача первого свободного в read/write пуле, mutex/condition), методы закрытия и **снимок статуса** (`in_use`, `idle`, `waiters`), опционально **`max_wait_seconds`** на ожидание слота (у драйвера в конфиге — **`pool_max_wait_seconds`**).
- **`PostgreSQLDriver` (`postgres.py`):**
  - В `__init__` добавить, например, **`self._pool: Optional[PostgreSQLConnectionPool] = None`** (рядом с существующим `self.conn: Any` / аналогичной моделью).
  - **Инициализация пула в `connect()`, а не в `__init__`:** в этом проекте `import psycopg` выполняется внутри `connect()` (~133–139); пул создавать там же, после успешного импорта и `_connect_kwargs`. Учесть **`_reconnect_main`**: при потере основного `conn` логика пула и операций должна оставаться согласованной (отдельный подпункт в PR при необходимости).
  - **Где использовать пул:** только ветка **self-managed** в `execute` и `execute_batch` (замыкания `do_run`, использующие сейчас `self.conn`). Ветку с **`transaction_id in self._transaction_manager._transactions`** и выделенным **`conn`** из менеджера транзакций **не менять** — транзакция остаётся на одном соединении.
  - **`disconnect` (~279–293):** перед закрытием основного `self.conn` вызвать закрытие всех соединений пула (например **`self._pool.close_all()`**), согласованно с порядком «журнал → transaction_manager → соединения».
  - Публичный **`pool_status() -> dict`** (или типизированная структура) на драйвере — источник метрик для шага 6.
- Ровно **5** соединений **в пуле** self-managed к СУБД; **3** write, **2** read; потокобезопасность на уровне пула. Дополнительно драйвер держит основное `conn` и коннекты транзакций (см. раздел статуса аудита).
- Классификация **read vs write** — в драйвере / `PostgreSQLOperations`, не в универсальном клиенте.
- **Запрещено:** пул в `DatabaseClient` / `rpc_dispatch.py`.

### Шаг 2a — убрать глобальную сериализацию `InProcessRpcClient` после готовности пула

Любой оставшийся критический участок под lock обосновать в PR.

  `vectorization_worker_pkg/processing.py`, `processing_cycle.py`, `processing_cycle_projects.py`, `batch_processor.py`, `file_batch_packing.py` — и любые другие вхождения по репозиторию.

### Шаг 5 — согласованность транзакций и долгих операций с пулом

- **Файлы:** `postgres_transactions.py`, при необходимости операции/индексатор.
- Ветка RPC с **явным `transaction_id`** и соединением из **`_transaction_manager._transactions`** (см. [шаг 2](#implementation-map)) **остаётся** на выделенном `conn`; пул self-managed её не заменяет.
- Транзакция держит одно выданное соединение; `index_file` / длинные write — не блокируют **весь** пул без необходимости (документировать поведение при исчерпании трёх write-conn).

<a id="step-6-pool-observability"></a>

### Шаг 6 — observability

- Метрики пула: занято/свободно по **write** (из 3) и **read** (из 2), счётчики **ожидающих** потоков на lane (`waiters` в `snapshot()` / `pool_status()`), таймаут ожидания слота (`max_wait_seconds`, из конфига драйвера **`pool_max_wait_seconds`**).
- **`get_database_status`:** в `get_database_status_build.build_database_status_result(db, ...)` аргумент **`db`** — это **`DatabaseClient`** (`get_database_status.py` ~69–73). У клиента **нет** публичного `db._driver`. На практике для Postgres in-process: **`getattr(db.rpc_client, "handlers", None)`** → **`handlers.driver`**, затем вызов **`pool_status()`** если драйвер — `PostgreSQLDriver` и пул внедрён. Альтернатива (если не хочется трогать тип `Any` в билдере): тонкий публичный метод на `DatabaseClient` / отдельный RPC — выбрать по минимальности диффа и стилю проекта.
- Поля ответа в духе `pg_write_pool_in_use`, `pg_read_pool_in_use` (имена зафиксировать в PR); для не-Postgres или socket-клиента — не заполнять или нули.

### Шаг 7 — тесты (фокус: Postgres in-process)

- Wire / протокол: тесты на **`RPCRequest.to_dict` / `from_dict`** с полем **`priority`** (см. шаг 1).
- Нагрузочный сценарий: конкурентные write + read; read (аналог `list_projects`) не блокируется на время длительного write при наличии свободного read-conn.
- Unit на выдачу «первое свободное» в пределах пула 3+2.
- Integration по возможности: фон (`update_indexes` и т.п.) + `list_projects` < 1–2s при приёмке.
- Тесты SQLite subprocess на новую политику — **не** блокируют фазу 1; перенос в [фазу 2](#phase-2-sqlite).

## Фаза 2: SQLite (только после ворот)

**Цель:** выровнять семантику «интерактив не голодает из-за фона» с фазой 1 для режима **SQLite subprocess**, не ломая парадигму: политика остаётся в **`RPCServer` / `RequestQueue`**, не в универсальном `DatabaseClient`. Схему **3 write + 2 read к одному файлу БД** **не** переносить буквально — у SQLite другая физика; своя модель соединений/очереди в **специфическом** sqlite-рантайме.

**Предпосылка:** пройдены [ворота фазы 1](#phase-1-gate).

### Шаг S-1 — учёт `priority` из wire JSON (при необходимости)

- **Файлы:** `rpc_protocol.py` (если в фазе 1 отложили wire), `rpc_server.py` (разбор и передача в очередь).
- Согласовать с фактическим JSON-RPC: приоритет из запроса воркера → уровень `RequestPriority` или отдельная lane-логика внутри subprocess.

### Шаг S-2 — расширить `_priority_for_request`

- **Файл:** `code_analysis/core/database_driver_pkg/rpc_server.py`.
- Правила в духе фазы 1: `index_file` / пакеты воркера → более низкий приоритет относительно интерактивных MCP-запросов (конкретные уровни enum — в задаче).

### Шаг S-3 — starvation guard в `RequestQueue` (по желанию продуктовой политики)

- **Файл:** `code_analysis/core/database_driver_pkg/request_queue.py`.
- Согласовать с тем, что в фазе 1 для PG заложен guard в диспетчере; поведение двух движков не обязано бит-в-бит, но не должно противоречить документированной семантике.

### Шаг S-4 — тесты и приёмка SQLite

- Регрессия socket-транспорта; сценарии конкуренции MCP vs воркеров на SQLite (по аналогии с интеграцией фазы 1).
